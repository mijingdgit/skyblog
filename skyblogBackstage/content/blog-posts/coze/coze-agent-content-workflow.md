## 智能体先解决流程问题

很多人做智能体时，会一开始就追求“全自动写文章”。但对技术博客来说，更值得自动化的不是替你写完所有内容，而是把重复流程规范下来：选题、收集资料、生成提纲、检查结构、整理发布清单。

扣子的优势在于工作流编排直观，可以把一个大任务拆成多个节点，每个节点只做一件明确的事。

## 工作流目标

这个智能体的目标不是输出最终成稿，而是输出一份可写作的准备材料：

1. 选题是否适合当前博客定位。
2. 文章面向谁，解决什么问题。
3. 提纲是否有完整的起承转合。
4. 需要补充哪些代码、截图或链接。
5. 发布前检查哪些字段。

这样生成物更容易被人工校正，也更适合长期维护。

## 输入设计

推荐让用户输入四类信息：

```text
主题：Vue3 文章列表页优化
目标读者：有基础的前端学习者
已有材料：项目中 ArticleListView.vue 的实现
文章目标：解释筛选、排序、搜索和路由同步
```

输入越结构化，智能体越不容易发散。如果只输入“帮我写一篇 Vue3 博客”，模型会倾向于生成通用内容，和项目实践的贴合度会下降。

## 节点拆分

可以把工作流拆成 5 个节点：

1. 选题判断：判断主题是否适合博客栏目。
2. 资料摘要：从用户提供的材料中提炼事实。
3. 提纲生成：输出二级标题和每节要点。
4. 风格校正：统一语气、长度和技术深度。
5. 发布清单：生成后台发布时要填写的字段。

每个节点都要限制输出格式。比如“发布清单”可以固定为：

```json
{
  "title": "文章标题",
  "excerpt": "140 字以内摘要",
  "category": "Vue3",
  "tags": ["Vue3", "TypeScript", "API"],
  "is_featured": false
}
```

格式稳定之后，后续接 API 或人工复制都会轻松很多。

## 提示词要强调来源

内容类智能体最容易犯的错误是补充不存在的细节。可以在节点提示词里明确写：

```text
只能基于用户提供的项目材料生成提纲。
如果某个实现细节材料中没有出现，请标记为“待补充”，不要自行编造。
输出要包含“已确认信息”和“待确认信息”两部分。
```

这能让智能体从“写作机器”变成“整理助手”。对技术内容来说，可靠比流畅更重要。

## 与博客后台结合

如果博客后台已经提供文章 API，可以进一步把工作流输出整理成草稿字段：

1. `title`：文章标题。
2. `excerpt`：摘要。
3. `content`：Markdown 正文草稿。
4. `category`：分类。
5. `tags`：标签。
6. `is_published`：默认 false。

建议第一阶段只生成草稿，不要自动发布。技术文章需要人工确认代码、版本和结论，自动发布的风险不值得。

## 质量检查清单

发布前可以让智能体检查：

1. 标题是否具体。
2. 摘要是否说明读者收益。
3. 每个二级标题是否有实际信息。
4. 代码块是否有语言标识。
5. 是否存在“显然、简单、众所周知”这类空泛表述。
6. 结尾是否给出可执行的下一步。

这些检查不复杂，但能显著提高文章成品的稳定性。

## 小结

扣子智能体适合做内容流程的“整理台”。让它负责拆解、归纳、检查和生成结构化草稿，人负责判断、补充事实和最终发布。这样的分工更稳，也更适合技术博客长期更新。
